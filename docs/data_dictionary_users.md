# Data Dictionary — Users Table

**Table name:** `users`  
**Owner:** Imanzi Beni  
**Engine:** InnoDB  
**Purpose:** Stores every distinct person or organisation that appears 
in a MoMo transaction — senders, recipients, agents, and merchants.

---

## Columns

| Column | Data Type | Nullable | Default | Description |
|---|---|---|---|---|
| user_id | INT | No | AUTO_INCREMENT | Surrogate primary key. An autogenerated integer that uniquely identifies each user. Never reused. |
| full_name | VARCHAR(120) | No | — | Full name of the person or organisation, written exactly as it appears in the SMS message body. |
| phone | VARCHAR(20) | Yes | NULL | Phone number stored as text, not as a number. May be masked e.g. *********013 or carry a country prefix e.g. 250791666666. Cannot be used in arithmetic. |
| user_type | ENUM | No | CUSTOMER | Classifies the role of the user. CUSTOMER is an ordinary MoMo user. AGENT is a registered MoMo agent. MERCHANT is a business or shop. OWNER is the holder of the account from which the SMS data was exported. |
| created_at | DATETIME | No | CURRENT_TIMESTAMP | The exact date and time when this user record was first inserted into the database by the ETL pipeline. |

---

## Constraints

| Constraint | Column | Rule |
|---|---|---|
| PRIMARY KEY | user_id | Uniquely identifies each row |
| NOT NULL | full_name, user_type, created_at | These fields must always have a value |
| DEFAULT 'CUSTOMER' | user_type | If no type is specified the pipeline defaults to CUSTOMER |
| DEFAULT CURRENT_TIMESTAMP | created_at | Automatically set to insertion time |

---

## Relationships

| Related table | How | Rule |
|---|---|---|
| transaction_participants | users.user_id → transaction_participants.user_id | One user can appear in many participant rows (1:N). A user cannot be deleted if participant rows still reference them (ON DELETE RESTRICT). |
